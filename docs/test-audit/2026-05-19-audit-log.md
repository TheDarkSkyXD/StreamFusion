## U0 — Playwright → electron-mcp playbook migration

The Playwright spec runner (14 `.spec.ts` files, fixtures, page objects, `playwright.config.ts`) was deleted in favor of the existing `debug-electron-mcp`-driven playbook suite. Rationale: Playwright was never wired into CI, the build-target path had drifted (`out/StreamFusion-<platform>-x64/` vs electron-builder's `release/`), and the project uses `debug-electron-mcp` for interactive E2E in practice. The two specs without playbook counterparts — `app-launch.spec.ts` and `navigation.spec.ts` — were rewritten as `00-app-launch.playbook.md` and `00b-sidebar-navigation.playbook.md`. `@playwright/test`, `@axe-core/playwright`, and the `test:e2e*` scripts were dropped from `apps/desktop/package.json`. `apps/desktop/scripts/inspect.ts` (a one-off Playwright-based CDP probe) was removed; `debug-electron-mcp` covers the same surface.

Smoke verification: `npm start` launched the app, `debug-electron-mcp` reached the window on the project's registered port (9231 / `streamforge-monorepo`), and the new `00-app-launch.playbook.md` pass criteria were observed (title "StreamFusion", sidebar present, no fatal exceptions).

## Follow-up: TwitchHermesClient stop()-during-CONNECTING — 2026-05-20

`apps/desktop/src/backend/services/chat/twitch-hermes-client.ts` `stop()` was calling `this.ws.close()` unconditionally during effect cleanup. Under React StrictMode dev double-invoke (and any prod fast unmount / channelId-switch before the WebSocket handshake completes), the browser fires its spec-required "WebSocket is closed before the connection is established" console error because `close()` on a `CONNECTING` socket runs the WHATWG "fail the WebSocket connection" path.

**Fix shape:** new module-private helper `closeWebSocketSafe(ws)` detaches `onmessage`/`onerror`/`onclose`, then branches on `readyState`:
- `CLOSING`/`CLOSED`: null `onopen` and return.
- `CONNECTING`: install a deferred-close `onopen` lambda that closes the socket once the handshake lands; also install a minimal `onerror` that nulls the deferred lambda so a failed handshake releases the captured WebSocket promptly.
- `OPEN`: null `onopen` and close immediately (original behavior).

`stop()` now delegates the close path to the helper. Other `ws.close()` callsites in this file (`resetPongTimer` at line 163, `handleMessage("reconnect")` at line 200) are unchanged — both fire only after `handleWelcome`, when the socket is `OPEN`, so they never hit the CONNECTING-close case.

**Regression tests added:** 5 in `apps/desktop/tests/backend/services/chat/twitch-hermes-client.test.ts` under the new `TwitchHermesClient lifecycle (start/stop)` describe:
- `stop()` during CONNECTING defers close, emits only `"connecting"` then `"disconnected"` on the consumer state bus, and does NOT emit a spurious `"connected"` when the deferred close fires after `_open()`.
- Hung-handshake failure path: `onerror` nulls the deferred-close `onopen` so the captured WebSocket is collectable.
- `stop()` after the socket is OPEN closes immediately (regression guard for unchanged behavior).
- `stop()` while readyState is CLOSING is a no-op (does not double-close).
- `stop()` while readyState is CLOSED is a no-op (does not double-close).

The describe block uses a local `MockWebSocket` stub (mirrors the queueMicrotask-based onclose dispatch from the EventSub mock at `tests/backend/api/platforms/twitch/twitch-eventsub-client.test.ts`) and `vi.stubGlobal("WebSocket", MockWebSocket)` per the tests/AGENTS.md backend-stub idiom. The mock + helper duplication across the two test files is acknowledged tech-debt; extraction to `tests/helpers/mock-websocket.ts` is a separate refactor.

**Source-diff-revert verification:** the first lifecycle test (deferred-close on CONNECTING) was authored against the pre-fix `stop()` body and `expect(ws.closeCallCount).toBe(0)` fails as expected (`closeCallCount` is `1`). After applying the `closeWebSocketSafe` helper, all 5 lifecycle tests pass. Verified on commit `e2141ea` (initial fix) and on the follow-up commit landing the onerror release path + audit-log entry.

**Outstanding follow-ups (surfaced by /ce-code-review against e2141ea, NOT landed here):**
- **TwitchEventSubClient has the identical unguarded `close()` gap** at `apps/desktop/src/backend/api/platforms/twitch/twitch-eventsub-client.ts` ~line 220. Same fix shape applies; either share `closeWebSocketSafe` via `apps/desktop/src/backend/shared/` or duplicate the guard inline. Tracked as a separate follow-up.
- **Pre-existing `onclose`-clobber race** at `twitch-hermes-client.ts:123` (in `connect()`'s `onclose` handler): server-initiated reconnect or pong-timeout close can clobber `this.ws` after `scheduleReconnect()` has assigned a new socket. Fix shape is a generation counter captured in the `onclose` closure. Pre-existed e2141ea; tracked separately.
- **Rapid channelId-thrash behavior shift:** the deferred-close path holds the half-open socket for handshake duration (typically 50-300ms) where the pre-fix immediate-close initiated TCP teardown right away. Under stress this can accumulate concurrent in-flight sockets against Twitch's undocumented per-IP Hermes connection limits. Mitigation is consumer-side (TwitchChat useEffect debounce or per-client concurrent-CONNECTING cap), not in the helper. Verify empirically under rapid channel-switching before deciding the mitigation shape.

## Follow-up: Kick image-fetch network-down latch — 2026-05-20

`apps/desktop/src/backend/api/platforms/kick/kick-client.ts` `fetchImageBytes()` short-circuited to null whenever `isNetworkLikelyDown()` returned true (lines 376-378 pre-fix). The `kick-image://` custom protocol handler at `apps/desktop/src/backend/protocols/kick-image-protocol.ts:58` translated null into HTTP 404, which fired `<img>.onerror` in the renderer, which latched the broken state in `ProxiedImage`'s `hasError` flag (`apps/desktop/src/components/ui/proxied-image.tsx:99,124`). A single 3-second unhealthy window — rolled forward by concurrent `net::ERR_FAILED` bursts from other Kick callers — therefore left Kick avatars/thumbnails stuck on the fallback initial until the host component remounted.

**Fix shape:** removed the `isNetworkLikelyDown()` short-circuit from `fetchImageBytes` only (other Kick callers — `request()`, `stream-endpoints.ts:_doFetchPublicStreamBySlug` — still gate on the flag because they have retry budgets and benefit from the back-off). The `acquireKickRequestSlot` semaphore (max 4 concurrent) already caps the thundering-herd risk the gate was guarding against. Companion change: added a `timeoutMs` parameter to `electronRequestBinary` (default 15000) and pass `3000` from the image path so during a sustained outage the discover-page grid resolves to broken-image fallbacks in ~40s rather than ~195s with the API timeout.

**Accepted tradeoffs (documented inline in the bypass-justification comment):**
- Image `net::ERR_*` failures now feed `recordTransientNetworkError`, so a CDN-only outage (Kick images down, API up) can arm the gate for other Kick callers. Surfaced by /ce-code-review as a possible cross-domain leak; accepted because eliminating the cross-feed requires either a separate health-tracker for images or a flag plumbed through `recordTransientNetworkError`, both of which expand scope without a concrete bug to fix.
- During outages, image fetches contend with API calls for the shared 4-slot semaphore. The 3s image timeout caps the worst-case starvation window.

**Regression test added:** `apps/desktop/tests/backend/api/platforms/kick/kick-client-image-fetch.test.ts` — stubs `electronRequestBinary` via `vi.spyOn` on the kickClient singleton, asserts the spy is called exactly once when `isNetworkLikelyDown()` is mocked true (the gate-bypass contract) and that the result flows back as bytes (the success-path contract). Direct boundary assertion — does not depend on Electron's net/session module shape or vitest's require/import interop.

**Source-diff-revert verification:** restored the `if (isNetworkLikelyDown()) return null;` block at line 376 of `kick-client.ts`. Test failed with `expected "electronRequestBinary" to be called 1 times, but got 0 times` — proving the spy assertion is sensitive to the gate's presence. Removed the block; test passed. Verified on current HEAD; diff range is the deletion of the 3-line gate from `fetchImageBytes` plus the `timeoutMs` parameter addition on `electronRequestBinary`.

**Suite delta:** 165 files / 1216 tests → 165 files / 1217 tests (+1 test; all green).
